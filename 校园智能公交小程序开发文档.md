# 校园智能公交系统小程序开发文档

## 1. 项目概述

### 1.1 项目名称
校园智能公交系统（微信小程序）

### 1.2 项目背景
本项目用于校园场景下的公交出行信息服务，面向学生、司机、管理员三类角色，提供车辆实时位置、预计到站时间、座位占用情况、车内温度等信息，提升通勤效率与乘车体验。

由于是课程大作业，系统中的车辆运动、乘客上下车、温度变化等数据均采用模拟数据驱动；地图展示可对接腾讯地图组件实现“拟真实时运动”效果。

### 1.3 建设目标
- 为学生提供“上车前 + 上车后”全流程可视化乘车信息。
- 为司机提供位置上报、空调/温度调节、到站播报控制能力。
- 为管理员提供账号、车辆、路线、站点、排班等基础管理能力。
- 实现可演示、可扩展、结构清晰的课程级完整软件工程作品。

## 2. 角色与功能需求

## 2.1 角色划分
1. 学生端
2. 司机端
3. 管理员端

## 2.2 学生端功能

### 2.2.1 上车前
- 查看当前所有运行公交车位置（地图 + 列表）。
- 查看“当前站点”下一辆车预计到达时间（ETA）。
- 查看各车辆座位占用程度（如 20/40、颜色进度条）。
- 查看各车辆温度信息（例如 24°C）。

### 2.2.2 上车后
- 选择自己在车内的座位（座位图点选）。
- 将座位占用状态同步给其他学生查看。
- 查看当前到达站点。
- 查看后续各站预计到达时间。

## 2.3 司机端功能
- 车辆位置上报（真实GPS或模拟轨迹点）。
- 车内空调开关与目标温度调节。
- 到站播报（手动触发/自动触发）。
- 座位占用概览（辅助司机了解满载情况）。

## 2.4 管理端功能
- 创建司机账号、重置密码、启停账号。
- 添加公交车（车牌、核载人数、设备编号等）。
- 规划路线（站点顺序、方向、首末班时间）。
- 配置线路车辆与司机排班。
- 查看运行监控看板（在线车辆、延误、满载预警）。

## 3. 技术方案（小程序核心）

## 3.1 技术栈建议
- 前端：微信小程序原生（WXML + WXSS + JS）
- UI：自定义组件 + 简易图表（进度条/状态卡）
- 地图：微信 map 组件（可叠加 marker、polyline）
- 后端（建议二选一）
  - 方案A：Node.js + Express + MySQL（本地/云服务器）
  - 方案B：微信云开发（CloudBase：云函数 + 云数据库）
- 实时通信：WebSocket（车辆位置、温度、座位变化推送）

> 大作业建议：优先采用微信云开发，部署快、接入简单、演示稳定。

## 3.2 架构分层
- 表现层（小程序页面）：学生端/司机端/管理端页面
- 业务层：ETA计算、座位分配、车辆状态机、播报逻辑
- 数据层：车辆、站点、线路、司机、用户、座位占用、轨迹点
- 推送层：WebSocket事件通道（位置更新、站点到达、状态广播）

## 3.3 模块结构建议
- `pages/student/*`：学生端页面
- `pages/driver/*`：司机端页面
- `pages/admin/*`：管理端页面
- `pages/common/login`：统一登录
- `components/bus-card`：车辆信息卡
- `components/seat-grid`：座位网格
- `components/eta-list`：站点ETA列表
- `utils/mock/*`：模拟器（车辆运动、温度、上下客）
- `services/*`：接口请求封装

## 6. 接口设计（示例）

## 6.1 认证
- `POST /api/auth/login`
- `GET /api/auth/profile`

## 6.2 学生端
- `GET /api/buses/active` 获取运行车辆列表
- `GET /api/stations/{id}/next-bus-eta` 获取下一辆车ETA
- `GET /api/buses/{id}/seats` 获取车辆座位状态
- `POST /api/buses/{id}/seats/select` 学生选择座位
- `GET /api/buses/{id}/eta-list` 获取该车后续站点ETA

## 6.3 司机端
- `POST /api/driver/buses/{id}/location` 上报位置
- `POST /api/driver/buses/{id}/ac` 调整空调状态与温度
- `POST /api/driver/buses/{id}/broadcast` 触发到站播报

## 6.4 管理端
- `POST /api/admin/drivers` 创建司机账号
- `POST /api/admin/buses` 添加车辆
- `POST /api/admin/routes` 创建路线
- `POST /api/admin/schedules` 创建排班

## 6.5 WebSocket事件
- `bus.location.updated`
- `bus.temperature.updated`
- `bus.seat.updated`
- `bus.station.arrived`

## 7. 核心算法与业务逻辑

## 7.1 ETA（预计到达时间）
简化公式：
- `ETA = 剩余路径距离 / 当前平均速度`
- 站点ETA列表可用“累加站间距离 / 速度”计算

增强版：
- 加入站点停靠时长（如每站 20~40 秒）
- 加入早晚高峰速度系数

## 7.2 座位选择冲突控制
- 采用“乐观锁 + 二次校验”机制：
  1. 前端点击座位提交
  2. 后端检查座位状态是否仍为 `available`
  3. 成功则写入用户ID并广播
  4. 失败则返回“已被占用”，前端刷新座位图

## 7.3 到站判定与播报
- 通过车辆与站点的距离判定：
  - 距离 < 100m：触发“即将到站”
  - 距离 < 30m：触发“到站”
- 播报支持自动触发与手动补播。

## 8. 公交运动模拟方案（重点）

## 8.1 模拟目标
在没有真实校车硬件与GPS设备条件下，模拟车辆“连续移动 + 到站停靠 + 速度变化 + 状态变化”。

## 8.2 模拟数据来源
- 方式1：预置路线轨迹点（推荐）
  - 每条路线存储一组 `lat/lng` 点（polyline）
  - 定时推进当前位置索引
- 方式2：站点插值生成轨迹点
  - 根据相邻站点坐标做线性插值，生成若干中间点

## 8.3 模拟运行机制
- 定时器每 2~3 秒推进一次位置：
  - `currentPointIndex += step`
  - 更新 `lat/lng/speed/heading`
- 到站后停靠 20~40 秒再继续
- 每次位置变化通过 WebSocket 广播给学生端

## 8.4 状态联动模拟
- 载客量：随机波动或按站点规则增减
- 温度：围绕目标温度做小范围随机浮动（±1°C）
- 座位：可预置“随机占座乘客”增强演示效果

## 8.5 腾讯地图/GPS模拟展示
可行方案：
1. 小程序 map 组件中展示：
   - `markers`：车辆和站点
   - `polyline`：路线轨迹
2. 使用模拟轨迹点逐帧更新车辆 marker 坐标，实现运动动画效果。
3. 如使用真实手机调试，可在开发者工具中开启位置模拟或在真机以测试定位回传。

> 结论：无需真实公交，即可在地图上做出“接近实时行驶”演示效果。

## 9. 权限与安全设计

## 9.1 权限控制
- 基于角色的访问控制（RBAC）：
  - 学生仅可访问学生端接口
  - 司机仅可操作分配给自己的车辆
  - 管理员可访问管理接口

## 9.2 数据安全
- 登��态使用 token（或云开发登录态）
- 关键接口校验角色与资源归属
- 记录关键操作日志（如改路线、改排班）

## 11. 测试与验收标准

## 11.1 功能测试
- 学生端可看到车辆动态位置、ETA、座位与温度
- 学生可成功选择座位并同步给他人
- 司机可控制空调温度并触发播报
- 管理员可创建司机、车辆、路线

## 11.2 业务一致性测试
- 车辆到站后当前站点自动推进
- ETA随车辆位置变化而更新
- 座位冲突场景下仅一人占座成功

## 11.3 演示验收
- 地图上至少 2~3 辆车同时模拟运行
- 展示一条完整线路从起点到终点流程
- 三端都可登录并完成关键操作

## 12. 课程答辩演示建议

建议准备一段 5~8 分钟演示流程：
1. 管理员创建司机、车辆并绑定路线
2. 司机端启动行程，开始位置上报
3. 学生端查看车辆靠近站点与下一班ETA
4. 学生上车后选座，其他同学端实时看到占座变化
5. 司机调整温度并触发到站播报
6. 展示系统统计与运行状态页面

## 14. 总结
本方案以微信小程序为核心，构建“学生-司机-管理员”三端联动的校园智能公交系统。在大作业条件下通过模拟数据驱动车辆运动与状态变化，并结合地图可视化、实时推送和座位管理机制，实现完整、可演示、可迭代的软件系统。
